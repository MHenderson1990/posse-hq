# Posse HQ

Group event planner for a fixed friend group ("the Posse"). Calendar-first iOS app:
everyone sees the crew's events in one place, RSVPs, coordinates in per-event threads,
and gets events synced to their Apple Calendar.

**This README is the source of truth for scaffolding and building the project. Do not
improvise features beyond it. Anything ambiguous or listed under "Proposed" must be
confirmed with Mo before building.**

---

## Platform & Stack (confirmed)

- **Target:** iOS App Store app
- **Client:** React (Vite) wrapped with **Capacitor** — NOT React Native/Expo
- **Backend:** Node.js / Express / MongoDB (same stack as Mo's prior apps)
- **Native layer:** Capacitor plugins for push notifications and EventKit calendar access
- **User group:** all-iPhone friend group; Android is out of scope for v1
- **Code style:** prefer `let` over `const`

## v1 Features (confirmed)

### Calendar — the main screen
- **Month view, custom-built grid** (do not use a calendar library) modeled on Apple
  Calendar: date number top-right of each cell, adjacent-month days dimmed, today
  marked with a gradient-filled circle.
- **Single-day events:** flat category-colored slivers with truncated titles,
  max 2 per cell, then a "+N more" indicator.
- **Multi-day events: true spanning bars** — one continuous category-colored bar per
  week row, wrapping across week rows with squared-off continuation edges
  (Apple-style). Implemented as an absolutely-positioned lane overlay per week row.
- **Tap-a-day panel** below the grid: full event cards for the selected day with
  category stripe, title, time, location, comment count, RSVP avatars/counts, and
  an "I'm in" RSVP button.
- Reference implementation of the exact look and interaction:
  `event-planner-calendar-mockup-v3.html` (approved by Mo).

### Design system (confirmed)
- **Light theme default + optional dark theme** — per-user setting
  (light / dark / follow system) stored on the user profile.
- **Blue-purple gradient** `#3B4EFF → #6C3DF4 → #9D4EDD` used ONLY on: app header,
  today marker, primary buttons. Everything else is flat.
- **Flat category colors** on slivers, spanning bars, and card stripes. Starting
  palette (groups can define their own categories/colors):
  - Trips `#0496FF` · Food `#FF9F1C` · Games & Chill `#8B5CF6` · Parties `#FF2E7E` ·
    Activities `#2EC4B6`
- Overall vibe: colorful, bold, friendly. Rounded type for headers (Nunito or similar).

### Events
- Anyone in the group can create an event (no admin/host role).
- Event fields: title, date(s), start/end time, location, category, description.
- Multi-day events are first-class (see spanning bars above).

### RSVPs
- Three states: **going / maybe / can't make it**.
- Event cards show avatar row + "N going · N maybe" counts.

### Per-event comment threads
- Each event has a comment/notes thread for coordination ("who's bringing what").
- **No standalone group chat in v1** — deliberately descoped; threads only.

### Push notifications (APNs via Capacitor push plugin + backend sender)
Notify group members ONLY for these four things — nothing else:
1. Someone RSVPs to an event
2. Someone comments on an event
3. A new event is created
4. Upcoming-event reminder (backend cron job, e.g. ~24h before start)

### Apple Calendar integration — EventKit (confirmed for v1)
Two features riding on one calendar permission grant:
1. **Write sync:** events the user RSVPs "going" to are written into their Apple
   Calendar and kept updated/deleted on changes. Store the created EventKit event
   IDs on-device per app event for reconciliation; reconcile when the app wakes
   (push notifications are the wake trigger).
2. **Availability ("N of M are free"):** the app reads the user's calendar
   **on-device only** and reports **free/busy blocks only** to the server — never
   event titles, locations, or details. This privacy boundary is a hard requirement
   and must be stated in the permission prompt UX.

### Theming/infrastructure requirements
- Registration + login; group membership.
- All data is scoped by `groupId` from day one (design supports multiple groups
  later; UI can assume one group for v1).

## Data models (starting point)

- **User**: name, email, auth, themePreference, groupIds
- **Group**: name, inviteCode
- **Event**: title, groupId, startDate, endDate, startTime, endTime, location,
  categoryId, createdBy, description, recurrenceRule (null if one-off),
  status (poll | scheduled)
- **DatePollOption**: eventId, candidateDate, votes[userId]
- **BringItem**: eventId, label, claimedBy (nullable)
- **Rsvp**: eventId, userId, status (going/maybe/no)
- **Comment**: eventId, userId, text, createdAt
- **Category**: groupId, name, color
- **DeviceToken**: userId, token, platform
- **FreeBusy**: userId, date, busyBlocks[] (start/end only — no details)

---

### Group joining, polls, recurring, bring lists (confirmed)
- **Invite-code group joining** — members join the group with a code, so new people
  can be added to the Posse later. No friend-request system.
- **Date polls** — an event can start as a poll: creator proposes 2–3 candidate
  dates, group votes, creator locks the winning date to finalize the event.
- **Recurring events** — weekly/biweekly/monthly repeat options (e.g. game night
  every other Friday).
- **Bring list** — optional per-event list of items (potluck/BYOB); members claim
  items in the event view.

## Explicitly v2 / out of scope for v1

- Standalone group chat (threads only in v1)
- Android support
- Google Calendar API integration (rejected — OAuth verification burden)
- Subscribable iCal feed / .ics Add-to-Calendar button (redundant with EventKit;
  feed is the fallback if someone declines calendar permission)
- Multiple groups per user in the UI (schema supports it; UI doesn't need it)
- Photo albums, expense splitting, public event discovery
